# TaskAI: Wiki Feature + Postgres Migration Implementation Plan

## Context

TaskAI is a task management system currently using SQLite. The user wants to add collaborative wiki functionality as a **secondary feature** while keeping tasks as the primary product. This requires migrating from SQLite to Postgres to support advanced full-text search and real-time collaboration.

**Why this change:**
- **Wiki need:** Projects need rich documentation alongside tasks
- **Real-time collaboration:** Multiple users editing docs simultaneously via Yjs CRDT
- **Search capability:** Fast full-text search across wiki content
- **Postgres migration:** SQLite FTS insufficient; Postgres provides better FTS, concurrency, and production scalability

**Goals:**
- ✅ Keep all existing task features working
- ✅ Zero downtime migration to Postgres
- ✅ Add collaborative wiki (Yjs + WebSocket)
- ✅ Full-text search with Postgres FTS
- ✅ Expose wiki via MCP tools

## High-Level Strategy

**7-Phase Incremental Approach** (14-22 days total):

1. **Phase 0-1:** SQLite → Postgres migration (3-5 days)
2. **Phase 2:** Wiki REST API & schema (2-3 days)

Each phase is independently testable with clear rollback procedures.

## Technology Decisions

### Database
- **Postgres Driver:** `github.com/jackc/pgx/v5/stdlib` (handles query placeholders automatically)
- **Migration:** Custom Go tool to copy data from SQLite to Postgres
- **Why Postgres:** Superior FTS (tsvector + GIN indexes), better concurrency for WebSockets

### Real-time Collaboration
- **WebSocket:** `github.com/gorilla/websocket` (Go standard)
- **CRDT:** Yjs (client-side) + Node.js microservice for server-side processing
- **Why Node.js microservice:** No mature Go Yjs library; pragmatic to use JavaScript-native Yjs in separate service

### Search
- **Postgres FTS:** Built-in `tsvector` with generated columns
- **Trigram:** `pg_trgm` extension for fuzzy title autocomplete
- **Indexing:** GIN indexes on `search_vector` column

## Database Schema Changes

### New Postgres Tables (Migration 024)

```sql
-- Wiki pages
CREATE TABLE wiki_pages (
    id BIGSERIAL PRIMARY KEY,
    project_id BIGINT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    title VARCHAR(500) NOT NULL,
    slug VARCHAR(500) NOT NULL,
    created_by BIGINT NOT NULL REFERENCES users(id),
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE(project_id, slug)
);

-- Yjs updates (append-only log)
CREATE TABLE yjs_updates (
    id BIGSERIAL PRIMARY KEY,
    page_id BIGINT NOT NULL REFERENCES wiki_pages(id) ON DELETE CASCADE,
    update_data BYTEA NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    created_by BIGINT REFERENCES users(id)
);

-- Page snapshots
CREATE TABLE page_versions (
    id BIGSERIAL PRIMARY KEY,
    page_id BIGINT NOT NULL REFERENCES wiki_pages(id) ON DELETE CASCADE,
    version_number INTEGER NOT NULL,
    yjs_state BYTEA NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE(page_id, version_number)
);

-- Content blocks (for search)
CREATE TABLE wiki_blocks (
    id BIGSERIAL PRIMARY KEY,
    page_id BIGINT NOT NULL REFERENCES wiki_pages(id) ON DELETE CASCADE,
    block_type VARCHAR(50) NOT NULL,
    level INTEGER,
    headings_path TEXT,
    canonical_json JSONB,
    plain_text TEXT,
    position INTEGER NOT NULL,
    search_vector tsvector GENERATED ALWAYS AS (
        setweight(to_tsvector('english', COALESCE(headings_path, '')), 'A') ||
        setweight(to_tsvector('english', COALESCE(plain_text, '')), 'B')
    ) STORED
);

CREATE INDEX idx_wiki_blocks_search ON wiki_blocks USING GIN(search_vector);
CREATE EXTENSION IF NOT EXISTS pg_trgm;
```

       // Parse request: query, project_id, limit, recency_days
       // SQL:
       SELECT wb.page_id, wp.title, wb.id, wb.block_type,
              ts_headline('english', wb.plain_text, websearch_to_tsquery($1)),
              ts_rank(wb.search_vector, websearch_to_tsquery($1)) as rank
       FROM wiki_blocks wb
       JOIN wiki_pages wp ON wp.id = wb.page_id
       WHERE wp.project_id = $2
         AND wb.search_vector @@ websearch_to_tsquery($1)
       ORDER BY rank DESC
       LIMIT $3
   }

## Rollback Plans

### Per-Phase Rollback

| Phase | Rollback Trigger | Action | Data Loss |
|-------|------------------|--------|-----------|
| 1 | Database errors, slow queries | Revert `DB_DRIVER` to `sqlite`, restart | None |
| 2 | Migration failure | Drop wiki tables, revert migration | Wiki only |
| 3 | WebSocket crashes | Disable WebSocket endpoint, use polling fallback | None |
| 4 | Yjs corruption | Disable snapshot/indexing workers | Delayed indexing |
| 5 | Search errors | Disable search endpoint | None |
| 6 | MCP integration issues | Revert MCP server version | None |
| 7 | Production outage | Revert Docker image, restore DB snapshot | Last N minutes |

### Emergency Procedures

**If production crashes:**
1. Immediately revert to last known good Docker image: `docker-compose up -d --build api:<previous-tag>`
2. Check logs: `docker-compose logs -f api | grep ERROR`
3. If database issue: Verify Postgres connection with `psql -h <host> -U taskai -d taskai -c "SELECT 1"`
4. Disable problematic feature: Set env var `WIKI_ENABLED=false`, restart

**If data corruption detected:**
1. Stop all writes: `docker-compose stop api`
2. Restore from latest backup: `pg_restore -U taskai -d taskai backup.dump`
3. Investigate root cause offline
4. Fix and redeploy

             PageID: pageID,
